# Техническое видение проекта (`vision.md`)

## Основа
Telegram-бот с LLM для сопровождения учебного потока: быстро объясняет термины/идеи, отвечает на вопросы и помогает сформулировать уточняющий вопрос.

## Технологии (включая зависимости, venv, сборка)
- Язык: `Python 3.x` (рекомендация: `3.11+`).
- Виртуальное окружение: `venv` в `.venv/`.

### Зависимости
- Управление зависимостями: `uv`.
- Принцип: воспроизводимые версии через `uv`-lock (например, `uv.lock`).
- Разделение: runtime-зависимости и dev-зависимости (линт/формат/тесты).

### LLM-клиент и провайдер
- Клиент: OpenAI-совместимый API-клиент (конфигурируется через `base_url`, чтобы ходить в OpenRouter).
- Провайдер: OpenRouter (ориентир).
- Аутентификация: ключ провайдера из переменных окружения (не из репозитория).
- Выбор модели/параметров (`model`, `temperature`, и т.п.) через конфиг.

### Telegram-бот
- Фреймворк: `aiogram`.
- Транспорт: `polling` (для MVP без вебхуков).

### Автоматизация
- `make` для команд:
  - `make install` — установка зависимостей через `uv`
  - `make run` — запуск бота
  - `make lint/format` — по согласованию (инструменты зафиксируем позже)

## Принципы разработки
- KISS: минимум сущностей, понятные границы слоёв, без “фреймворков ради фреймворков”.
- Разделение ответственности:
  - `telegram`-обработчики знают только про вход/выход Telegram (сообщения, клавиатуры, polling).
  - сервис/слой диалога формирует запрос к LLM (system-подсказка + минимальный контекст/история).
  - отдельный клиент LLM инкапсулирует вызовы OpenAI-compatible API (OpenRouter).
- Асинхронность: обработчики и вызовы к LLM без блокировок (async/await), чтобы не “подвешивать” polling.
- Состояние: хранить только то, что реально нужно для контекста диалога (история/последний “режим”), без внешней файловой памяти курса.
- Тестируемость: LLM-вызовы вынести в отдельный слой/класс, чтобы можно было подменять в unit-тестах.
- Обработка ошибок: таймауты/ошибки провайдера -> короткий ответ пользователю + подробности в логах.
- ООП: по правилу “один класс — один файл” для ключевых классов (клиент LLM, сервис диалога/сессии, Telegram-handlers). Остальное можно держать компактно по модулям.

## Структура репозитория
- Корень:
  - `README.md`
  - `Makefile`
  - `pyproject.toml`
  - `uv.lock`
  - `docs/` (например, `docs/idea.md`, `docs/vision.md`)
  - `.env.example` (пример переменных окружения, без секретов)
- `src/` (python “src-layout”):
  - `src/llmstart/` — пакет приложения
    - `config.py` — чтение настроек (env -> переменные)
    - `logging_setup.py` — настройка логирования
    - `llm_client.py` — клиент OpenAI-compatible API (OpenRouter)
    - `dialog_service.py` — сборка system-промпта, формирование запроса, вызов LLM
    - `state.py` — состояние сессии/контекста (для MVP: in-memory)
    - `telegram_handlers.py` — aiogram handlers (polling), клавиатуры/команды
    - `bot.py` — сборка бота (инициализация, запуск polling)
    - (по желанию) `main.py` — точка входа, чтобы `make run` было простым
- `tests/`:
  - тесты `dialog_service.py` с мокнутым `llm_client.py`

## Архитектура
- Слои/ответственности:
  - `telegram_handlers.py`: обработка входящих апдейтов Telegram и отправка текста обратно в чат. Тут нет бизнес-логики и работы с LLM.
  - `dialog_service.py`: бизнес-логика тьютора. Собирает контекст (system-подсказка + ограниченная история), вызывает LLM, формирует финальный текст для пользователя.
  - `llm_client.py`: адаптер над OpenAI-совместимым клиентом, чтобы ходить в OpenRouter (`base_url`, ключ, модель, параметры).
  - `state.py`: минимальное состояние диалога (контекст/история для MVP).
  - `bot.py`: сборка зависимостей (`config` -> `llm_client` + `state` + `dialog_service`) и запуск `polling`.

- Поток обработки сообщения:
  - handler получает текст пользователя/команду и достаёт сессию из `state`.
  - `dialog_service` формирует запрос к LLM (system + история + текущее сообщение) и запрашивает ответ.
  - `dialog_service` обновляет состояние (например, добавляет user/assistant сообщения) и возвращает готовый текст в `telegram_handlers`.
  - `telegram_handlers` отправляет ответ пользователю.

- Ошибки LLM (важно):
  - Ловим в `dialog_service` (например, таймауты, ошибки провайдера, rate limit).
  - Пользователю возвращаем внятное короткое сообщение (“Сейчас не могу ответить, попробуй позже” или “Слишком много запросов, подожди немного”).
  - Детали ошибки пишем в лог (для диагностики).

## Модель данных / состояние
- Цель: хранить минимальный контекст для диалога, без “памяти курса” (только история в рамках сессии).
- Сущность `Session` (в `state.py`, MVP):
  - `chat_id` — ключ сессии (получаем из Telegram)
  - `mode` — текущий режим/контекст (по умолчанию “тьютор для урока”)
  - `history` — список сообщений в формате для LLM: `[{role: "user"|"assistant", content: "..."}]`
  - `updated_at` (по желанию) — для контроля устаревания/сброса
- Ограничение истории (KISS):
  - держим последние `N` сообщений *или* обрезаем по лимиту длины/символов
  - при превышении лимита: старые сообщения отбрасываем (без “сложной суммаризации”)
- Хранение:
  - для MVP: in-memory (словарь `chat_id -> Session`)
  - БД/файлы — позже, когда появится необходимость переживать перезапуск
- Конкурентность:
  - если вдруг один чат шлёт много сообщений подряд, защищаем критическую секцию обновления истории (например, `asyncio.Lock` на сессию); для максимального KISS можно оставить упрощённо, если не увидим гонок

## Работа с LLM (API, провайдер)
- Цель: один простой путь “диалог → запрос к LLM → текст ответа”.
- Провайдер: OpenRouter.
- Клиент: OpenAI-compatible API-клиент.
- Настройки:
  - `base_url` (OpenRouter endpoint)
  - `api_key` (из env)
  - `model` и параметры вроде `temperature`
  - таймаут запроса (чтобы не зависать)
- Формирование запроса:
  - `system` сообщение задаёт роль/стиль тьютора (спокойный тьютор, объясняет простыми словами).
  - дальше `history` из `state` (последние реплики).
  - в конце текущее сообщение пользователя.
- Ответ:
  - берем текст из ответа LLM.
  - при необходимости делаем минимальную постобработку (например, чистка лишних служебных маркеров).
- Поведение:
  - при ошибках провайдера/таймауте возвращаем в сервис “ошибку с понятным типом”, чтобы `dialog_service` мог показать пользователю внятное сообщение.

## Сценарии работы пользователя
- `/start`: приветствие, короткая инструкция “задай вопрос по теме урока или попроси объяснить простыми словами”.
- Обычное сообщение (без команд): бот отвечает по роли тьютора и старается сделать ответ понятным новичку; при необходимости задаёт 1 уточняющий вопрос.
- Запрос на объяснение термина/идеи:
  - бот даёт определение простыми словами
  - добавляет 1–2 мини-примера
  - предлагает короткий “self-check”, чтобы пользователь мог понять, что действительно разобрался
- Запрос “как подойти к задаче / с чего начать”:
  - бот предлагает план шагов (чеклист)
  - указывает, какие данные/ограничения уточнить в условии
- Запрос “сформулируй вопрос преподавателю/в сообщество”:
  - бот превращает непонимание пользователя в готовый текст вопроса
  - включает: что уже пробовал, где застрял, что ожидаешь получить
- Команда `/reset`: очищает контекст/историю для `chat_id` и начинает диалог заново.
- Сбой LLM/провайдера:
  - пользователь получает внятное короткое сообщение без технических деталей
  - бот предлагает повторить позже

## Конфигурация и секреты
- Конфиг грузим из переменных окружения (через `src/llmstart/config.py`).
- Секреты не храним в репозитории:
  - `TELEGRAM_BOT_TOKEN` — токен Telegram
  - `OPENROUTER_API_KEY` — API-ключ провайдера
- Параметры LLM:
  - `LLM_BASE_URL` — base_url для OpenAI-compatible клиента (OpenRouter)
  - `LLM_MODEL` — имя модели
  - `LLM_TEMPERATURE` — температура по умолчанию (необязательно)
  - `LLM_TIMEOUT_SECONDS` — таймаут запроса
- Параметры бота:
  - `LOG_LEVEL` — уровень логирования
- Локальная разработка:
  - добавить `.env.example` с перечислением переменных (без значений)
  - `.env` не коммитить (отдельно указать в `.gitignore`)
- Логирование секретов:
  - никогда не логируем API-ключи (в конфиге/логгере не выводим их значения).

## Логирование
- Пишем логи в файл и в stdout.
- Уровни логов: `DEBUG/INFO/WARNING/ERROR`.
- Что логируем:
  - старт бота и итоговую конфигурацию (без секретов)
  - входящие апдейты Telegram (например, `chat_id`, тип события, длина текста)
  - вызовы LLM: модель, длительность, статус/ошибка (без ключей и без полного текста prompt’а)
  - ошибки `dialog_service`: тип ошибки/класс, таймауты/rate limit (technical details в логе)
- Формат: структурированный текстовый лог (timestamp + уровень + сообщение, при ошибках — stack trace).
- Ротация логов (при необходимости для MVP): ограничить размер/количество файлов.

## Сборка и запуск
- Требования: Python, доступ к интернету (для зависимостей и LLM), переменные окружения (см. `Конфигурация и секреты`).
- Разработческий запуск (через `make`):
  - `make install` — создаёт/обновляет `venv` и ставит зависимости через `uv`
  - `make run` — запускает Telegram-бот (polling)
- Управление окружением:
  - запускать из окружения, где доступны переменные (обычно через `.env`)
  - `make run` / приложение берут значения из env и передают в `config.py`
- Ошибки старта:
  - при отсутствии обязательных токенов/ключей показываем понятную ошибку в stdout и логируем `ERROR` в файл
- Линт/формат:
  - `make lint` и/или `make format` — по согласованию (инструменты выберем позже)

## Деплой на локальном Docker
- Добавляем `Dockerfile` (и при желании `docker-compose.yml`) для запуска бота как обычного CLI-приложения.
- Контейнер запускает polling и не требует входящих портов (в MVP).
- Перед запуском прокидываем переменные окружения через `--env-file` или `-e`:
  - `TELEGRAM_BOT_TOKEN`
  - `OPENROUTER_API_KEY`
  - `LLM_BASE_URL`, `LLM_MODEL` и прочие настройки
- Пример сценария запуска (в dev-команде/на ноутбуке):
  - `docker build -t llmstart-bot .`
  - `docker run --rm --env-file .env llmstart-bot`
